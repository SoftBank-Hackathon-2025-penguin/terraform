설계 문서를 기반으로 Terraform Workspace를 활용한 인프라 구성 파일을 작성하겠습니다.[1][2][3]

## Terraform 파일 구조

```
terraform/
├── main.tf                 # 메인 리소스 정의
├── variables.tf            # 변수 선언
├── outputs.tf              # 출력 값 정의
├── backend.tf              # S3 백엔드 설정
├── provider.tf             # AWS Provider 설정
├── vpc.tf                  # VPC 리소스
├── ec2.tf                  # EC2 리소스
├── dynamodb.tf             # DynamoDB 테이블
├── s3.tf                   # S3 버킷
├── lambda.tf               # Lambda 함수
├── cloudwatch.tf           # CloudWatch 설정
├── sns.tf                  # SNS 토픽
└── iam.tf                  # IAM Role 및 정책
```

### backend.tf
```hcl
terraform {
  backend "s3" {
    bucket               = "penguin-land-tfstate"
    key                  = "sessions/terraform.tfstate"
    region               = "ap-northeast-2"
    workspace_key_prefix = "sessions"  # 워크스페이스별 상태 파일 격리
    dynamodb_table       = "penguin-land-tfstate-lock"
    encrypt              = true
  }

  required_version = ">= 1.0"
  
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}
```

```hcl
              amazon-linux-extras enable corretto17
              yum install -y java-17-amazon-corretto-headless

              mkdir -p /var/log/penguin-land
              chown ec2-user:ec2-user /var/log/penguin-land
```

### lambda.tf
```hcl
# Lambda 함수용 ZIP 파일 준비
data "archive_file" "lambda_alert_processor" {
  type        = "zip"
  output_path = "${path.module}/lambda_alert_processor.zip"

  source {
    content  = <<-EOF
      import json
      import boto3
      
      def lambda_handler(event, context):
          """
          CloudWatch 알람을 받아서 처리하고 Spring Boot 백엔드로 전달
          """
          print(f"Received event: {json.dumps(event)}")
          
          # SNS 메시지 파싱
          for record in event['Records']:
              sns_message = json.loads(record['Sns']['Message'])
              alarm_name = sns_message.get('AlarmName', 'Unknown')
              new_state = sns_message.get('NewStateValue', 'Unknown')
              reason = sns_message.get('NewStateReason', 'No reason provided')
              
              print(f"Alarm: {alarm_name}")
              print(f"State: {new_state}")
              print(f"Reason: {reason}")
              
              # 여기서 백엔드 API 호출 가능 (예: HTTP POST)
              # requests.post('http://backend-url/api/alerts', json={...})
          
          return {
              'statusCode': 200,
              'body': json.dumps('Alert processed successfully')
          }
      EOF
    filename = "index.py"
  }
}

# Lambda 함수 (알람 중계)
resource "aws_lambda_function" "alert_processor" {
  filename         = data.archive_file.lambda_alert_processor.output_path
  function_name    = "${var.project_name}-alert-processor-${terraform.workspace}-${var.session_id}"
  role             = aws_iam_role.lambda_execution.arn
  handler          = "index.lambda_handler"
  source_code_hash = data.archive_file.lambda_alert_processor.output_base64sha256
  runtime          = "python3.11"
  timeout          = 30

  environment {
    variables = {
      ENVIRONMENT = terraform.workspace

    Name      = "${var.project_name}-alert-processor"

  }
}
```

```hcl
  alarm_actions             = [aws_sns_topic.alerts.arn]

  metric_query {
    id          = "e1"
    expression  = "ANOMALY_DETECTION_BAND(m1)"
    label       = "CPUUtilization (Expected)"
    return_data = true
  }

  metric_query {
    id          = "m1"
    return_data = true
    metric {
      metric_name = "CPUUtilization"
      namespace   = "AWS/EC2"
      period      = 300
      stat        = "Average"
      dimensions = {
        InstanceId = aws_instance.app_server.id
      }
    }
  }

  tags = {
    Name      = "${var.project_name}-cpu-anomaly"
    SessionID = var.session_id
  }
}
```

## 사용 방법

### 1. 초기 설정 (State 버킷 수동 생성)
```bash
# S3 버킷 생성 (Terraform State 저장용)
aws s3 mb s3://penguin-land-tfstate --region ap-northeast-2

# DynamoDB Lock 테이블 생성
aws dynamodb create-table \
  --table-name penguin-land-tfstate-lock \
  --attribute-definitions AttributeName=LockID,AttributeType=S \
  --key-schema AttributeName=LockID,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST \
  --region ap-northeast-2
```

### 2. Workspace 생성 및 배포[4][5]
```bash
# Terraform 초기화
terraform init

# 새로운 워크스페이스 생성 (세션별)
terraform workspace new session-abc123

# 또는 기존 워크스페이스로 전환
terraform workspace select session-abc123

# 워크스페이스 목록 확인
terraform workspace list

# 배포 계획 확인
terraform plan -var="session_id=abc123"

# 리소스 배포
terraform apply -var="session_id=abc123" -auto-approve

# 배포 진행 상태 확인
terraform show

# 출력 값 조회 (JSON 형식)
terraform output -json
```

### 3. 리소스 삭제
```bash
# 특정 워크스페이스 선택
terraform workspace select session-abc123

# 리소스 제거
terraform destroy -var="session_id=abc123" -auto-approve

# 워크스페이스 삭제 (리소스 제거 후)
terraform workspace select default
terraform workspace delete session-abc123
```

## 핵심 특징[6][7]

### Workspace 기반 격리
- 각 세션은 별도의 Terraform Workspace로 관리되어 완전히 격리됨[2]
- S3 State 파일은 `sessions/<workspace>/terraform.tfstate` 경로에 자동 저장[3]
- DynamoDB Lock으로 동시 실행 방지[1]

### 세션별 태그 전략
- 모든 리소스에 `SessionID` 태그를 자동 추가하여 사용자 구분[1]
- `Environment` 태그로 현재 Workspace 식별
- 리소스 정리 시 태그 기반 필터링 가능

### 모니터링 및 알림
- CloudWatch에서 CPU, 에러율, 레이턴시 등 하드코딩된 임계값 모니터링[1]
- Anomaly Detection으로 이상 징후 자동 탐지
- SNS → Lambda → Backend API 파이프라인으로 실시간 알림[1]

이 구조는 설계 문서의 요구사항을 충족하며, Terraform Workspace를 활용해 각 사용자 세션을 완전히 격리된 환경에서 관리합니다.[2][3][1]
